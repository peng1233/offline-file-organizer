# 定价档位 / 交付范围（模板，可直接复制）

> 目的：在 Upwork / Fiverr / Toptal / GitHub Sponsors 上快速给出“清晰、可验收、可回滚”的档位描述。
> 
> 注意：下面是**模板/建议措辞**，不是承诺。实际报价/周期/改稿次数以平台页面为准；且所有沟通与交付尽量留在平台内（合规）。

---

## 通用交付原则（建议固定写法）

- **离线优先**：工具可在本地浏览器离线运行，文件不上传服务器（隐私友好）。
- **不破坏原文件**：默认只生成新文件名方案与下载产物，不批量修改原始文件（除非明确要求）。
- **可回滚**：任何改动都提供回滚方式（恢复旧版本/撤销配置）。
- **验收可量化**：用“同一输入 → 输出对比 + 自检脚本输出”作为验收证据。

---

## 档位 A：快速支持（小需求 / 轻量定制）

适合：你已经能跑起来，只差一点点（文案/按钮/兼容性/说明）。

- 交付内容（示例，可改）：
  - 修 1 个小问题或加 1 个小功能（不重构、不大改结构）
  - 补 1 段说明文档（含截图/步骤）
  - 提供自检输出（例如 `node tools/self_check.js` / `node lite/selfcheck.js`）
- 验收标准：
  - 你给一组示例输入；我给修复前/后对比说明 + 自检 OK 输出
- 风险提示（建议固定）：
  - 不同浏览器对 file://、剪贴板、选择文件夹 API 权限不同，必要时会提供 localhost 启动方案

---

## 档位 B：可交付小版本（功能闭环）

适合：你要一个“能给同事/客户直接用”的小版本。

- 交付内容（示例，可改）：
  - 完成 1 个小闭环功能（例如：批量改名 → 导出 CSV → 下载 ZIP/逐个下载）
  - 补齐一页式交付说明（Quick start + 常见问题）
  - 产出可分发包（zip）与 SHA256（如项目已有构建脚本则复用）
- 验收标准：
  - 你按说明 1 次成功跑通；或我提供录屏/截图 + 自检输出

---

## 档位 C：集成 / 团队落地（流程与标准化）

适合：你要把它变成团队流程的一部分（可复用、可维护）。

- 交付内容（示例，可改）：
  - 增加“项目范围/验收标准/风险/回滚”模板并落地到仓库 docs
  - 增强自检覆盖（新增页面/脚本不会漏检）
  - 补兼容性说明（Windows/Chrome/Edge、localhost 与 file:// 区别）
- 验收标准：
  - 提供一份可复用的“交付清单”与最小闭环演示

---

## 平台字段速填（可选）

- **Fiverr Gig**：
  - Basic/Standard/Premium → 对应 档位 A/B/C
  - Requirements（客户需要提供）建议写：示例文件名/期望规则/目标浏览器/是否必须离线
- **Upwork Proposal**：
  - 先写 3 行价值点，再写 3 条验收标准，再给 1 个最小范围建议
- **GitHub Sponsors**：
  - 建议写成“Monthly support tiers”：
    - 小额：问题答疑/小修复
    - 中额：小功能闭环
    - 高额：路线图/优先级支持
